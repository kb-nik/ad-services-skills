# Operations

## Когда читать этот файл

- во время активного A/D раунда
- при подозрении на ongoing flag theft
- для preflight и health-диагностики
- при подготовке farm-ready exploit
- для проверки состояния `service closed`
- для incident response по ходу раунда

## Рабочее мышление

Сильной source-находки недостаточно в живом раунде.
Разделяй четыре состояния:

1. `source-suspected`
2. `locally-confirmed`
3. `live-confirmed`
4. `farm-ready`

Это не одно и то же.

## Лестница evidence

Используй максимально сильное подтверждение:

- reasoning по исходникам
- локальное воспроизведение
- совпадение с checker traffic
- live exploit proof
- повторяемый фарм по нескольким командам

Всегда явно помечай, какой уровень подтверждения у тебя есть.

## Когда флаги уже воруют

Переключайся из общего аудита в incident triage.

Вопросы:

- Какой именно сервис теряет флаги?
- Это подтверждено telemetry вроде `flag_out`, stream logs или повторяющимся фейлом чекера?
- Сервис всё ещё уязвим после последнего патча?
- Проблема в exploitability, availability или checker incompatibility?

Немедленные действия:

- подтвердить целевой сервис и время раунда
- связать входящий exploit traffic с предполагаемым путём атаки
- предпочесть узкую emergency mitigation широкому рефакторингу
- после emergency change обязательно повторно проверить легитимный checker flow

## Preflight и health

Считай preflight вспомогательным сигналом, а не доказательством эксплуатируемости.

Полезные сигналы:

- сервис up/down
- давление по диску
- статус docker compose
- NTP drift
- состояние monitoring/farm endpoints
- доступность опубликованных сервисов

Не путай:

- warning-level проблемы наблюдаемости
- с реальными exploit path
- и с корневой причиной кражи флагов

## Состояния сервиса

Явно классифицируй статус:

- `vulnerable`
- `closed`
- `unknown`
- `broken`
- `checker-broken`

Ставь `closed` только если выполнены оба условия:

- exploit path умер
- легитимное поведение сервиса осталось рабочим

## Жизненный цикл эксплойта

Эксплойт становится operational по ступеням:

1. primitive работает
2. путь до флага срабатывает один раз
3. путь совпадает с checker reality
4. эксплойт переживает шум сети и тайминги
5. эксплойт масштабируется на команды

Перед тем как называть эксплойт farm-ready, проверь:

- discovery targets
- retries и timeouts
- обработку дубликатов флагов
- логирование ошибок
- устойчивость к частичной деградации сервиса

## Traffic-guided reasoning

Если есть трафик или stream logs, используй их, чтобы понять:

- какие object names реально использует checker
- attackers читают или overwrite’ят
- эксплуатация идёт через HTTP, TCP, SSH, broker или worker
- attacker чистит артефакты после себя или нет

Очень часто трафик показывает реальный exploit path быстрее, чем статический ревью.

## Patch Map

При защите под временем строй patch map:

- файл
- диапазон строк
- класс бага
- минимальное изменение
- риск для пользователя
- команда или proof для проверки

Это помогает не править код хаотично и ускоряет reasoning про rollback.

## Как рапортовать по ходу раунда

Короткий статус должен отвечать на вопросы:

- какой сервис задет
- что подтверждено
- что пока только гипотеза
- текут ли флаги прямо сейчас
- задеплоен ли фикс
- проходит ли checker

Round reports должны быть короткими и stateful.
